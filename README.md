# halloween_dmx_pub

Public asset repository for Halloween DMX IoT devices based on the ESP32.  
Devices fetch **firmware binaries**, **SPIFFS filesystem images**, and **YAML
config files** from this repository over HTTPS.

---

## Repository layout

```
halloween_dmx_pub/
├── firmware/
│   ├── <device>/
│   │   └── v<major>.<minor>.<patch>/
│   │       ├── firmware.bin        # ESP32 OTA firmware image
│   │       └── meta.json           # Version metadata (date, notes, SHA-256)
│   └── <device>.manifest.json      # Firmware manifest — auto-generated by CI
├── spiffs/
│   ├── <device>/
│   │   └── v<major>.<minor>.<patch>/
│   │       └── spiffs.bin          # SPIFFS filesystem image
│   └── <device>.manifest.json      # SPIFFS manifest — auto-generated by CI
├── config/
│   └── <device>/
│       ├── config.yaml             # Device YAML configuration (no secrets)
│       └── fixtures/               # Fixture YAML profiles
└── .github/
    └── workflows/
        └── update-manifest.yml     # CI: regenerates both manifests on push
```

> **Firmware and SPIFFS have separate manifests.**  
> This allows them to be flashed independently, each with their own version history.

## Manifest format

Each manifest has the same flat structure:

```json
{
  "latest": "1.0.5",
  "versions": [
    {
      "version": "1.0.5",
      "date": "2026-03-01",
      "notes": "Release automatique",
      "url": "https://raw.githubusercontent.com/kubkpower/halloween_dmx_pub/main/firmware/<device>/v1.0.5/firmware.bin",
      "sha256": "ff6816d11adcc63025cb962ec14b77ab9bbd85edd8b35cbb4b3fb5281e5f1503"
    },
    {
      "version": "1.0.4",
      "date": "2026-03-01",
      "notes": "Release automatique",
      "url": "...",
      "sha256": "..."
    }
  ]
}
```

| Field | Description |
|---|---|
| `latest` | Version string of the most recent entry |
| `versions` | Array of all available versions, newest first |
| `version` | Semantic version string |
| `date` | Build date (ISO 8601), sourced from `meta.json` |
| `notes` | Release notes, sourced from `meta.json` |
| `url` | Direct HTTPS URL to the binary on `raw.githubusercontent.com` |
| `sha256` | SHA-256 digest of the binary (64 hex chars) |

The ESP32 device resolves the SPIFFS manifest URL from the firmware manifest URL
by replacing `/firmware/` with `/spiffs/`.

## Per-version `meta.json`

Each versioned directory under `firmware/<device>/v<x.y.z>/` contains a `meta.json`:

```json
{
  "version": "1.0.5",
  "date": "2026-03-01",
  "notes": "Release automatique",
  "firmware": { "sha256": "ff6816d1..." },
  "spiffs":   { "sha256": "d85e584c..." }
}
```

This file is generated by `scripts/publish_to_pub.ps1` in the private build repo
and is the authoritative source for `date` and `notes` in the generated manifests.

## GitHub Actions — `update-manifest.yml`

The workflow runs automatically on every push to `main` that touches a `.bin` file
or a `meta.json`.

**Steps:**
1. Checkout the full repository (`fetch-depth: 0`)
2. Run an embedded Python script which:
   - Scans all `firmware/<device>/v*/` directories
   - For each version: reads `meta.json` (UTF-8 BOM–tolerant), hashes all binaries
   - Writes `firmware/<device>.manifest.json` (firmware entries only)
   - Writes `spiffs/<device>.manifest.json` (SPIFFS entries only)
   - Entries sorted newest-first by semantic version
3. Commit and push with `git pull --rebase` before pushing (race-safe against concurrent triggers)

The commit message is `chore: update manifests [skip ci]` to prevent recursive triggers.

## Publishing a new version

Releases are created from the **private** `halloween_dmx` repository using:

```powershell
# 1. Bump version in platformio.ini:  -DFW_VERSION='"x.y.z"'
# 2. Run the publish script:
.\scripts\publish_to_pub.ps1
```

The script:
1. Compiles firmware (`esp32dev-pub`) and SPIFFS
2. Clones this repository
3. Writes `firmware/<device>/v<x.y.z>/firmware.bin`
4. Writes `spiffs/<device>/v<x.y.z>/spiffs.bin`
5. Writes `firmware/<device>/v<x.y.z>/meta.json` (date, notes, SHA-256 of both binaries)
6. Copies `.github/workflows/update-manifest.yml` to keep CI in sync with the private repo
7. Opens a pull request on this repository

After the PR is merged, the `update-manifest.yml` CI job regenerates both
manifests with correct SHA-256 hashes for all known versions.

## Adding a new device

1. Add the directory tree and binaries:
   ```
   firmware/<device>/v1.0.0/firmware.bin
   firmware/<device>/v1.0.0/meta.json
   spiffs/<device>/v1.0.0/spiffs.bin
   config/<device>/config.yaml
   ```

2. Push to `main`. The workflow auto-creates:
   - `firmware/<device>.manifest.json`
   - `spiffs/<device>.manifest.json`

3. Point your ESP32 firmware at:
   ```
   https://raw.githubusercontent.com/kubkpower/halloween_dmx_pub/main/firmware/<device>.manifest.json
   ```

## Version filtering on the ESP32 web UI

The Settings page in the device web UI filters the manifest version list
to display only relevant entries:

| Rule | Shown |
|---|---|
| Currently installed version | Always (highlighted) |
| 2 latest builds of the latest `major.minor` | e.g. 1.0.6 + 1.0.5 |
| Last build of the previous minor | e.g. last 1.0.x when on 1.1.x |
| Last build of the previous major | e.g. last 0.x.y when on 1.x.y |

This keeps the list compact as the number of versions grows.  
Firmware and SPIFFS are shown in separate tables, each compared to their
respective installed version.

## Security

### HTTPS

All URLs use `https://raw.githubusercontent.com`, which is TLS-protected.
ESP32 devices should verify the server certificate using the embedded
**DigiCert Global Root CA G2** (PEM format).

### Integrity verification

Devices **must** verify the `sha256` checksum after downloading a binary
and before flashing it. Reject the update if checksums do not match.

The `OtaManager` in the device firmware performs this automatically when
the manifest entry includes a `sha256` field.

### No secrets in this repository

Config files committed here must not contain passwords, API keys,
Wi-Fi credentials, or any other sensitive data.

## OTA manifest format

Each device type has a `firmware/<device-name>.manifest.json` that devices poll
to discover updates.  The manifest is validated against
[`manifest.schema.json`](manifest.schema.json).

```json
{
  "name": "halloween-dmx-controller",
  "chip": "esp32",
  "version": "1.2.0",
  "min_version": "1.0.0",
  "firmware": {
    "url": "https://raw.githubusercontent.com/kubkpower/halloween_dmx_pub/main/firmware/halloween-dmx-controller/v1.2.0/firmware.bin",
    "sha256": "<64-char hex digest>",
    "size": 983040
  },
  "spiffs": {
    "url": "https://raw.githubusercontent.com/kubkpower/halloween_dmx_pub/main/spiffs/halloween-dmx-controller/v1.2.0/spiffs.bin",
    "sha256": "<64-char hex digest>",
    "size": 262144
  },
  "config": {
    "url": "https://raw.githubusercontent.com/kubkpower/halloween_dmx_pub/main/config/halloween-dmx-controller/config.yaml",
    "sha256": "<64-char hex digest>"
  }
}
```

The `min_version` field enables **rollback protection**: devices running a
firmware version lower than `min_version` must update through intermediate
releases before applying this one.

## Automation – checksum updates

A GitHub Actions workflow (`.github/workflows/update-checksums.yml`) runs
automatically on every push to `main` that touches a `.bin` or `.yaml` file.
It executes `scripts/update_manifests.py`, which:

1. Scans `firmware/`, `spiffs/`, and `config/` for device directories.
2. Finds the highest semantic-version subdirectory for each device.
3. Computes the SHA-256 digest and file size for every binary/config.
4. Writes the results back into `firmware/<device>.manifest.json`.
5. Commits the updated manifests automatically.

To run locally:

```bash
python3 scripts/update_manifests.py
```

## Security

### HTTPS on ESP32

All URLs served by this repository use
`https://raw.githubusercontent.com`, which is TLS-protected.  
ESP32 devices **must** verify the server certificate to prevent
man-in-the-middle attacks.  Embed the **DigiCert Global Root CA G2**
certificate (PEM) in your firmware and pass it to `esp_http_client` /
`esp_https_ota`:

```c
// In your esp_https_ota_config_t:
.http_config = &(esp_http_client_config_t){
    .url            = MANIFEST_URL,
    .cert_pem       = digicert_global_root_g2_pem,  // include in your firmware
    .skip_cert_common_name_check = false,
},
```

Verify the active root CA and keep it up-to-date in your firmware:

```bash
openssl s_client -connect raw.githubusercontent.com:443 -showcerts 2>/dev/null \
  | openssl x509 -noout -text | grep "Issuer:"
```

Always embed the current root CA certificate and update it with each firmware
release.  GitHub's TLS certificate chain may change over time (e.g. during CA
rotations); verify the current root CA before shipping a new firmware build and
consider a fallback mechanism (e.g. a bundled Mozilla CA store) so that devices
remain updatable if the CA changes unexpectedly.

### Integrity verification

Devices **must** verify the SHA-256 checksum after downloading a binary and
before flashing it.  Reject the update if checksums do not match.

### No secrets in config files

YAML config files committed to this repository **must not** contain passwords,
API keys, Wi-Fi credentials, or any other sensitive data.  Use BLE provisioning
or a secure on-device key store for secrets.

### Memory considerations

- Stream large binaries directly into flash using `esp_https_ota` rather than
  buffering the entire image in RAM.
- Parse config YAML in a streaming/chunked manner; avoid loading the full file
  into the ESP32 heap at once.
- Keep SPIFFS images within the partition size defined in your partition table
  (typically 1–2 MB).
